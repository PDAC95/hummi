   -- 00000000000001_init_extensions_schemas.sql
   -- Phase 1 Plan 02 — extensions + schemas (INFRA-06, INFRA-07)

   CREATE EXTENSION IF NOT EXISTS pgcrypto WITH SCHEMA extensions;
   CREATE EXTENSION IF NOT EXISTS pg_cron WITH SCHEMA extensions;
   CREATE EXTENSION IF NOT EXISTS pg_net WITH SCHEMA extensions;

   CREATE SCHEMA IF NOT EXISTS ops;
   CREATE SCHEMA IF NOT EXISTS stripe;

   COMMENT ON SCHEMA ops IS 'Internal operations (audit_log, notifications_log, jobs)';
   COMMENT ON SCHEMA stripe IS 'Stripe webhook idempotency + cached state';
   ```

   Why `WITH SCHEMA extensions`: Supabase convention puts extensions into the dedicated `extensions` schema (already created by Supabase) instead of `public`, keeps `public` clean for app tables. pg_cron and pg_net are restricted-permission extensions on Supabase — only `postgres` role can install them, and the CLI runs as postgres so this works.

   ```markdown
   # Hummi Migration Runbook

   **Audience:** Anyone touching the DB schema. Follow these steps for every schema change.

   ## Policy (INFRA-02 — enforced)

   - **Never** edit schema directly in Supabase Studio for prod. All changes go through migration files.
   - Migration files live in `supabase/migrations/` and are timestamped: `YYYYMMDDHHMMSS_description.sql`.
   - The CLI assigns the timestamp via `supabase migration new <description>`.

   ## Local dev workflow

   1. `supabase migration new add_users_table` — creates an empty timestamped file
   2. Edit the file with your SQL (use `IF NOT EXISTS`, `IF EXISTS`, idempotent constructs)
   3. `supabase db reset` — drops + re-runs all migrations locally (Docker)
   4. Test against local. Iterate.

   ## Pushing to staging

   ```powershell
   $env:SUPABASE_ACCESS_TOKEN = "<token>"
   supabase link --project-ref <STAGING_PROJECT_ID>
   supabase db push --linked --dry-run   # always dry-run first
   supabase db push --linked             # apply
   ```

   ## Pushing to prod

   **Do not push manually.** Open a PR → merge to `main` → GitHub Actions `deploy-prod.yml` workflow (created in Plan 03) handles the push.

   ## Recovering from a bad migration

   - If a migration was applied to staging and is wrong: write a new migration that reverses it. Never edit an applied migration.
   - If a migration was applied to prod and is wrong: same. Never rewrite history.
   - To check what's applied: `supabase migration list --linked`.

   ## CLI version pin

   - Required: ≥ 2.84 (older versions had broken `--dry-run`, see RESEARCH.md Pitfall 1)
   - CI pin: see `.github/workflows/pr-checks.yml` `supabase/setup-cli@v1` `with: version:` field
   ```
