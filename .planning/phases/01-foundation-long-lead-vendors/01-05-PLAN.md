1. **Create `scripts/pre-commit.sh`** (RESEARCH.md §"Code Examples → Pre-Commit Hook"):
   ```bash
   #!/usr/bin/env bash
   # scripts/pre-commit.sh
   # The body of .git/hooks/pre-commit. Installed by scripts/install-pre-commit.sh.
   # Enforces INFRA-05: gitleaks scan + project-specific secret-leak grep.

   set -euo pipefail

   # 1. Gitleaks on staged content (fast, scoped to staged files only — see RESEARCH.md Pitfall 5)
   if command -v gitleaks >/dev/null 2>&1; then
     gitleaks git --pre-commit --staged --redact --no-banner || {
       echo ""
       echo "gitleaks: secret detected in staged changes. Commit blocked."
       echo "If this is a false positive, add a narrow allowlist entry to .gitleaks.toml."
       exit 1
     }
   else
     echo "WARNING: gitleaks not installed. Skipping secret scan."
     echo "  Install: scoop install gitleaks   (or download from https://github.com/gitleaks/gitleaks/releases)"
   fi

   # 2. Project-specific secret-leak grep (created in Plan 03)
   if [ -f scripts/secret-leak-grep.sh ]; then
     bash scripts/secret-leak-grep.sh || exit 1
   else
     echo "ERROR: scripts/secret-leak-grep.sh not found. Plan 03 must complete before this hook can run."
     exit 1
   fi

   exit 0
   ```

2. **Create `scripts/install-pre-commit.sh`**:
   ```bash
   #!/usr/bin/env bash
   # scripts/install-pre-commit.sh
   # Run once per dev machine after `git clone`. Idempotent.
   set -euo pipefail

   HOOK_DIR="$(git rev-parse --git-dir)/hooks"
   HOOK="$HOOK_DIR/pre-commit"

   if [ ! -d "$HOOK_DIR" ]; then
     echo "ERROR: not a git repo (no .git/hooks dir). Run from repo root."
     exit 1
   fi

   cp scripts/pre-commit.sh "$HOOK"
   chmod +x "$HOOK"

   echo "Pre-commit hook installed at: $HOOK"
   echo ""
   echo "Prerequisites for the hook to work:"
   echo "  - gitleaks installed: scoop install gitleaks (Windows) or brew install gitleaks (mac)"
   echo "  - ripgrep installed: scoop install ripgrep (Windows) or brew install ripgrep (mac)"
   echo ""
   echo "Test with a fake secret:"
   echo "  echo 'sk_live_FAKEFAKEFAKEFAKEFAKE12345678' > /tmp/test-secret.txt"
   echo "  git add /tmp/test-secret.txt && git commit -m 'test'"
   echo "  (should be blocked)"
   ```

3. **Mark both executable in git**:
   ```powershell
   git update-index --chmod=+x scripts/pre-commit.sh
   git update-index --chmod=+x scripts/install-pre-commit.sh
   ```

# Secrets Hygiene Runbook

**Owner:** Founder
**Last updated:** {today's date}

## Rules (non-negotiable)

1. Service-role keys (Supabase, Stripe secret, Resend, Twilio account SID + auth) live in:
   - GitHub Actions secrets (CI)
   - Supabase Edge Function env vars (runtime)

4. If a secret reaches git history (despite hooks): rotate the key, document the rotation, do NOT rewrite history.

## Rotated Keys (running log)

| Date | Vendor | Key type | Redacted leaked value | Action taken |
|------|--------|----------|-----------------------|--------------|
| YYYY-MM-DD | Stripe | publishable | pk_live_REDA... | Rotated, new key in GH Actions + Supabase Edge env |

## Rotation procedure (per vendor)

### Supabase
1. Dashboard → Project Settings → API → "Reset" anon or service_role key
2. Update `.env.local` (anon)
3. Update GitHub Actions secret `SUPABASE_ACCESS_TOKEN` if rotated (service_role rarely rotated; anon is what the SPA uses)
4. Redeploy Edge Functions: `supabase functions deploy --project-ref <ref>` (so new key is bundled with secrets)

### Stripe
1. Dashboard → Developers → API keys → Reveal secret key
2. Roll the secret key → confirm
3. Update GH Actions secret `STRIPE_SECRET_KEY`
4. Update Supabase Edge Function secret: `supabase secrets set STRIPE_SECRET_KEY="sk_live_..."`

### Resend
1. Dashboard → API Keys → revoke + create new
2. Update Supabase Edge Function: `supabase secrets set RESEND_API_KEY="re_..."`

### Twilio
1. Console → Account → API keys → revoke + create new
2. Update Supabase Edge Function: `supabase secrets set TWILIO_AUTH_TOKEN="..."`

### Sentry
1. Settings → Auth Tokens → revoke + recreate with same scopes
2. Update GH Actions secret `SENTRY_AUTH_TOKEN`

## Why we don't rewrite git history

A rotated key in history is inert — the value is no longer valid anywhere. Rewriting history requires every clone (including CI runners, future contributors) to force-pull, which is error-prone. The rotation IS the fix; documentation IS the evidence.
